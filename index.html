Fmod Integration

For this project I tasked myself with integrating Fmod to give our game a better soundscape. I wanted to integrate it in a way that would allow the other programmers to easily play sounds of all types through just one Method call. To do this I made a SoundManager that allows anyone to call a method passing an Fmod event reference and an enum for what type of sound it is. When the method is called the SoundManager takes care of the sound instancing and keeps references for the sounds played to lessen loading times for sounds that would be replayed later through storing them in a dictionary using the GameObject and Fmod event reference as a key, doing it this way also allowed for several objects to play the same sounds.

I also made saving functionality for the sound settings using a binary formatter. I would save the volume profile for each sound bank for Fmod and save that to a file, as well as saving a version file to make sure the right saving format was used to avoid issues when updating the games settings file format.

Enemy AI

The enemy AI in the game was originally very simple and only had a few different actions it could take, I expanded on this by making the enemies actions and ai a lot more customizable by giving them more actions they can do, and by allowing for modifications that can be done using conditions to modify their behaviors and what they can and cannot do. This then changes up the enmies available actions depending on conditions like, what other enemies are on the board with them, wether or not the player had already beaten the game before, is the enemy low on health and a lot of other conditions

AI Tools

To make the enemy AI a lot more easily customizable I decided to create a tool for the game designer to use when creating new enemies for the game. This tool allowed the designer to decide what conditions had to pass before an enemy could consider doing an action, then it would choose an action from all the possible actions depending on action priority and weight. All the conditions could also be used to modify other aspects of the enemies decision making, like modifying an actions likelyhood of happening by changing the weight of an action under certain circumstances, or even changing an actions strength value. There are also descriptive tooltips for all aspects of the tool to help the designers understand how it works

Enemy Actions and Conditions

All the actions in the game were made using the same template class with different methods that were predefined so making new actions for the enemies would be as quick as possible. The only thing that had to be programmed per action was the logic for who the enemy targets and what happens when the enemy performs their action.

The conditions for an enemies action would be checked through a separate class called the condition checker to see if all the actions conditions are met. This class has different ways of checking conditions, it can look through an array of actions then evaluate if the action is possible based its conditions, if the action is possible it is added to a list and eventually this list is returned containing all the current possible actions. Alternatly conditions can be checked with an array of conditions which is useful for other cases, like in our game where the conditions are also used to check if any action strength or weight modifications should be made.
